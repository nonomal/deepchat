# Provider Runtime Scope

## User Need

DeepChat needs OpenAI Codex OAuth support and a safer way for developers to add providers without
turning provider integration into a large dynamic runtime system.

## Goal

Deliver one runtime provider addition, `openai-codex`, and one repo-scoped developer workflow,
`add-provider`, that generates provider integration changes against the existing DeepChat provider
architecture.

## What This Round Adds

- `openai-codex` as a fully separate provider from `openai`, with its own `apiType`, auth
  lifecycle, routes, settings panel, adapter, and tests.
- OpenAI Codex OAuth sign-in with main-process token handling.
- A repo skill at `.agents/skills/add-provider/SKILL.md` for developer-assisted provider additions.
- A provider addition checklist that maps requested provider details to the current DeepChat files.
- Focused tests for Codex OAuth, Codex request routing, and the add-provider skill workflow.

## What This Round Leaves Out

- No `ProviderRuntimeDefinition` type.
- No generated runtime manifest.
- No `models.dev` sync pipeline.
- No full OpenCode provider parity target.
- No automatic package installation for provider SDKs.
- No renderer-side dynamic provider execution.
- No broad provider credential migration for existing API-key providers.
- No new special providers beyond `openai-codex`.

## Design Principles

- `PublicProviderConf` remains the source for provider display data, model metadata, icons, pricing,
  and capabilities.
- Provider execution continues to use existing DeepChat surfaces:
  - `DEFAULT_PROVIDERS`
  - `providerRegistry.ts`
  - `providerFactory.ts`
  - provider-specific classes where behavior exceeds an existing transport
  - typed routes and renderer clients
- Provider additions are explicit code changes generated by a repo skill and reviewed in normal PRs.
- Runtime behavior is selected from known transports, not inferred from upstream package names.
- Provider credentials and OAuth tokens stay in the main process for new OAuth flows.
- `openai-codex` keeps provider records, settings state, route payloads, credential storage, provider
  instances, and provider factory branches separate from the standard `openai` API-key provider.

## OpenAI Codex Provider Requirements

`openai-codex` must:

- Appear as a distinct provider choice from standard OpenAI API-key profiles.
- Use `apiType: openai-codex` and a dedicated provider implementation even when the outbound request
  is shaped as a Responses API request.
- Use an experimental flag and an environment kill switch.
- Support browser PKCE sign-in.
- Browser PKCE sign-in opens a DeepChat-owned authorization window.
- Support a device-code fallback when browser callback cannot be used.
- Persist OAuth credentials in OS-backed encrypted storage when available.
- Expose auth status to the renderer without raw tokens.
- Refresh tokens before expiry and coordinate concurrent refreshes.
- Route requests through a dedicated Codex adapter.
- Build the Codex model picker from the OpenAI provider database using the current Codex
  recommended model IDs: `gpt-5.5`, `gpt-5.4`, `gpt-5.4-mini`, and
  `gpt-5.3-codex-spark`.
- Replace stale persisted Codex model snapshots with the dedicated Codex runtime catalog during
  model refresh.
- Use `gpt-5.5` as the default Codex connection-check model.
- Include OpenAI Responses `instructions` on every Codex chat request. The value comes from the
  leading system prompt when present and falls back to a minimal DeepChat assistant instruction.
- Add ChatGPT backend routing headers, including the full `ChatGPT-Account-ID` when available and
  `OAI-Product-Sku: codex`.
- Skip `openai-codex` during provider-db background refresh events because its visible model list is
  served by the dedicated runtime catalog. Manual model refresh remains available from the Models
  tab.
- Preserve streaming, abort behavior, proxy behavior, tool calls, reasoning, and structured output
  where the upstream endpoint accepts them.
- Surface subscription or entitlement failures as provider errors without leaking tokens or account
  identifiers.

Implementation must verify current OpenAI Codex OAuth and endpoint constants during the task. Any
constant copied from OpenCode is treated as compatibility data, not as a stable public API.

## Add Provider Skill Requirements

`add-provider` must:

- Live in `.agents/skills/add-provider/SKILL.md`.
- Trigger when a developer asks Codex to add a DeepChat provider.
- Ask for or derive the minimum required inputs:
  - provider ID
  - display name
  - API type or known transport family
  - default base URL
  - auth type
  - model metadata source
  - test model and check strategy
  - docs/API-key/model URLs
  - request quirks such as headers or endpoint rewrites
- Generate changes against existing files instead of producing a runtime manifest.
- Refuse unsupported dynamic SDK installation as a default path.
- Prefer OpenAI-compatible transport for compatible HTTP APIs.
- Require an explicit special-provider implementation when auth, request shape, model discovery, or
  streaming behavior differs from existing transports.
- Include a verification checklist and test targets in its output.

## Acceptance Criteria

- `openai-codex` can be enabled without modifying existing `openai` provider profiles.
- Codex browser OAuth works against mocks.
- Codex device OAuth works against mocks.
- Codex token refresh is single-flight and persists rotated refresh tokens.
- Renderer route payloads for Codex auth contain no access token, refresh token, ID token, or raw
  account ID.
- Codex requests use the dedicated adapter and can be disabled by the kill switch.
- Codex requests include Codex backend auth headers and produce readable errors for rejected
  bad-request responses.
- Codex Responses requests include `instructions`, including flows without a configured system
  prompt.
- Codex model listing exposes the current recommended Codex models from provider-db metadata
  instead of filtering only model IDs containing `codex`.
- Codex model refresh replaces old `gpt-5-codex` and `gpt-5.1-codex` style snapshots with the
  current dedicated runtime catalog.
- The Models tab exposes a refresh action so users can update provider models from the model list
  surface.
- Provider-db loaded or updated events refresh already materialized DB-backed providers and skip
  `openai-codex`; users can manually refresh Codex models when they need a catalog update.
- Browser sign-in creates an internal authorization window and closes it when auth completes,
  fails, is cancelled, or times out.
- `openai-codex` has separate provider ID, `apiType`, config persistence, route namespace,
  credential storage, settings panel, and adapter from `openai`.
- Existing OpenAI, GitHub Copilot, custom OpenAI-compatible, Azure, Vertex, Bedrock, Anthropic,
  Gemini, and local-provider behavior remains compatible.
- `.agents/skills/add-provider/SKILL.md` exists and documents a concrete provider-addition workflow.
- The add-provider skill output maps to the current provider architecture and does not introduce a
  runtime manifest.
- No `ProviderRuntimeDefinition` or generated provider runtime files are added.
- SDD files contain no unresolved clarification markers.

## Non-Goals

- Provider parity with all `models.dev` providers.
- Automatic provider inference from `models.dev` or OpenCode package metadata.
- Runtime loading of arbitrary npm provider packages.
- Renderer-owned credential storage.
- Migration of all existing API keys into a new credential store.
- Replacing `PublicProviderConf`.
- Replacing the current settings provider page.
- Adding xAI OAuth, DigitalOcean OAuth, Snowflake Cortex, GitLab Duo, SAP AI Core, Cloudflare
  Gateway, Bedrock Mantle, or Vertex Anthropic in this round.

## Risks

- OpenAI Codex compatibility endpoints may change independently of DeepChat.
- OAuth credential storage adds a new security-sensitive path.
- Device/browser OAuth UI must avoid exposing transient secrets.
- Accidental reuse of OpenAI API-key provider helpers could couple Codex OAuth behavior to standard
  OpenAI profiles.
- The existing provider summary route currently exposes API-key fields to the renderer; this round
  must avoid extending that behavior to Codex OAuth credentials.
